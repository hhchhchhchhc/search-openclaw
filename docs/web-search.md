# Web Search

## OpenClaw 的 Web Search 为什么这么关键

讨论结论很明确：搜索配置会显著影响 Agent 查资料、核实信息和跟踪时效信息的能力。

配置对了，OpenClaw 才更像“会搜、会判、会引用”的执行代理。

很多人第一次遇到效果不理想时，会下意识怀疑模型本身。但在真实使用里，常见瓶颈其实是：

- 没有联网搜索，Agent 只能依赖已有上下文
- 搜索结果噪音高，找不到可信来源
- 单一搜索源结果偏科，容易漏掉关键资料
- 搜索层缺少结构化输出，Agent 很难稳定利用

对 Agent 来说，`Web Search` 带来的价值不是“可以搜一下”，而是下面这些更关键的能力：

- 获取时效信息
- 做多来源交叉验证
- 快速定位延伸阅读材料
- 给后续分析、总结、引用提供可靠输入

如果你要让 OpenClaw 做的是研究、资讯监控、内容整理、竞品分析、选题挖掘，那么 Web Search 往往是最先值得优化的一层。

## 默认优先

有卡且追求兼容性，优先 `Brave Search API`。

它的优势通常体现在：

- 社区验证更多
- 结果结构对 Agent 更友好
- 相对稳定
- 资料更多，踩坑成本更低

## 零门槛优先

不想绑卡，优先 `Tavily`。

它更适合：

- 刚开始尝试 OpenClaw 搜索配置
- 想先快速跑通一条可用链路
- 对接入复杂度更敏感

## 进一步降成本

如果基本搜索已经能跑通，可以进一步考虑：

- `Exa`
- `You.com`
- `iFlow`
- 多搜索源组合

这些方案未必是最省事的，但在一些场景下更适合继续压低成本，或者补足语义搜索和容灾能力。
